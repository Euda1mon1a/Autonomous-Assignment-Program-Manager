# Schema Drift Report

> Documenting known mismatches between SQLAlchemy ORM models and actual PostgreSQL schema.

**Date:** 2026-02-19
**Branch:** `feature/empty-table-features`
**Discovery:** Found during database seeding work

---

## Summary

Several resilience-related tables have **column name mismatches** between the SQLAlchemy model definitions in `backend/app/models/resilience.py` and the actual database columns. This occurs because the models were updated after the original migration was applied, without a corresponding follow-up migration.

---

---

## Impact

- **ORM `select()` queries** on affected tables will fail with `UndefinedColumnError` because SQLAlchemy generates SQL referencing model columns that don't exist in the database.
- **Seed scripts** must use raw SQL (`text()`) to bypass ORM column mapping.
- **API routes** that use these models may also fail if they attempt full-model queries.

---

## Recommended Resolution

Create an Alembic migration to align the database schema with the current model definitions:

```bash
cd backend
alembic revision --autogenerate -m "align_resilience_schema_with_models"
```

> [!CAUTION]
> Carefully review the autogenerated migration before applying. It may attempt to drop/rename columns with existing data. Manual editing of the migration will likely be needed.

---

## Workarounds in Use

1. **Seed scripts** use `text()` raw SQL for all inserts/queries on affected tables.
2. **API routes** for resilience features may need similar treatment until the alignment migration is applied.
3. **CHECK constraints** are documented in [SEEDING_GUIDE.md](SEEDING_GUIDE.md) for reference when inserting data.
