## PR 1316 — `feat: convert 11 constraints from hard to penalty variables`

### Decision
Request changes.

### Findings

1. High: four of the converted faculty constraints still target the resident assignment matrix instead of the faculty matrix, so their new CP-SAT penalties do not apply to actual faculty decisions.
   - `FacultyPrimaryDutyClinic` and `FacultyDayAvailability` still read `variables["template_assignments"]` in `backend/app/scheduling/constraints/primary_duty.py:247` and `backend/app/scheduling/constraints/primary_duty.py:543`, while the solver stores faculty decisions in `variables["faculty_template_assignments"]` in `backend/app/scheduling/solvers.py:1066-1070`.
   - `FacultyRoleClinic` and `SMFacultyNoRegularClinic` have the same problem and also look up faculty through `context.resident_idx` in `backend/app/scheduling/constraints/faculty_role.py:134` and `backend/app/scheduling/constraints/faculty_role.py:400`.
   - Repro: with only faculty vars populated, all four constraints add `0` objective terms. In a mixed resident+faculty context, `FacultyPrimaryDutyClinic` attaches penalty variables to the resident `x` variable because it uses faculty index `0` against the resident assignment map.

2. High: `FMITMandatoryCall` is still a no-op in solver behavior.
   - The new penalty path checks for Friday/Saturday overnight call vars in `backend/app/scheduling/constraints/fmit.py:388-435`.
   - The solver never creates Friday/Saturday `call_assignments`; it only creates Sun-Thu call vars in `backend/app/scheduling/solvers.py:326-341` and `backend/app/scheduling/solvers.py:955-970`.
   - The validator is still a placeholder that always returns satisfied in `backend/app/scheduling/constraints/fmit.py:493-496`.
   - Repro: in a solver-style context with an FMIT week and no Fri/Sat call vars, `FMITMandatoryCallConstraint.add_to_cpsat()` appends `0` objective terms.

3. Medium: the passing test story for this PR does not exercise the new CP-SAT behavior, so it misses both bugs above.
   - `backend/tests/scheduling/test_faculty_role_constraint.py:227` and `backend/tests/scheduling/constraints/test_primary_duty.py:210` are validate-only.
   - `backend/tests/scheduling/constraints/test_overnight_call.py:80` is init/eligibility/validate-only.
   - `backend/tests/scheduling/test_fmit_constraint.py:442-453` explicitly accepts the placeholder `FMITMandatoryCall` validator.
   - Result: targeted suites pass while the converted solver paths remain broken.

## PR 1318 — `feat: centralize call/FMIT calendar policy to DB settings`

### Decision
Request changes.

### Findings

1. High: the new calendar policy cache creates ghost state and inconsistent behavior across runtime paths.
   - `load_from_settings()` stores policy in module globals and catches all exceptions in `backend/app/scheduling/calendar_policy.py:39-66`.
   - The engine is the only production caller of `load_from_settings()` in `backend/app/scheduling/engine.py:1172-1174`.
   - Other production paths, including call coverage reporting in `backend/app/services/call_assignment_service.py:579-587` and `backend/app/services/call_assignment_service.py:792-796`, read the helper directly without ever loading DB settings.
   - Worse, when a prior run has loaded custom values, a later DB failure or missing settings row does not reset the cache to defaults; it silently keeps the old values. Synthetic repro on this PR: start with defaults, load `{1}`/`2`, then call `load_from_settings()` with a failing DB or `None` settings row; the module still returns `{1}` and `2`.
   - Impact: policy can depend on unrelated earlier requests in the same process instead of the current database state.

2. Medium: the PR does not actually make the new settings human-editable through the existing settings API/UI, despite the PR body claiming admin-UI tunability.
   - The model and migration add `overnight_call_weekdays` and `fmit_week_start_weekday` in `backend/app/models/settings.py:84-90` and `backend/alembic/versions/20260314_cal_policy_cols.py:18-36`.
   - But the settings schemas exposed to the API stop at the older fields in `backend/app/schemas/settings.py:6-45`.
   - A repo-wide search on this branch finds no frontend or API schema references to the new fields.
   - Result: the values are DB-backed, but not actually configurable through the app surface this PR references.

3. Medium: the backward-compatibility export `OVERNIGHT_CALL_DAYS` is now stale after settings load.
   - `call_coverage.py` binds `OVERNIGHT_CALL_DAYS = get_overnight_call_weekdays()` once at import time in `backend/app/scheduling/constraints/call_coverage.py:41`.
   - After `load_from_settings()` updates the helper state, `OVERNIGHT_CALL_DAYS` keeps the original default set while `is_overnight_call_day()` uses the new values.
   - I verified this directly: after loading `[4]`, `get_overnight_call_weekdays()` returned `{4}` while `OVERNIGHT_CALL_DAYS` remained `{0,1,2,3,6}`.
   - Runtime logic mostly uses the helper now, so this is not the primary blocker, but it leaves the exported constant inconsistent and will confuse tests and any downstream imports that rely on it.
